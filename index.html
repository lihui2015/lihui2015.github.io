整洁代码笔记

一、命名规则

1. 名副其实
public List<Cell> getFlaggedCells(){
    List<Cell> flaggedCells = new ArrayList<Cell>();
    for (Cell cell:gameBoard)
        if(cell.isFlagged())
            flaggedCells.add(cell);
    return flaggedCells;
}

注：变量、函数或类的名称应该告诉你，它为什么存在，它做什么事，应该怎么用。
如果名称需要注释来补充，那就不算是名副其实。

2. 避免误导

  • 应当避免使用与本意相悖的词。

    注：不要使用专有名词作为变量名

  • 别用accountList来指一组账号，除非它真的是List类型。

    注：可以用accountGroup替换

  • 提防使用不同之处较小的名词

  • 避免拼写前后不一致

  • 避免使用误导性名称

    注：l和o看起来像是‘壹’和‘零’。

3. 做有意义的区分

光是添加数字系列或是废话远远不够, 如果名称必须相异，那其意思也应该不同。 。

  • 反面案例：以数字系列命名（a1、a2，......aN）
  • 反面案例：以废话命名（product，productInfo，productData名称虽然不同，意思却无区别）

    Info和Data就像a、an和the一样，是意义含混的废话。

4. 使用读得出来的名称
  • 反面案例：定义了一个变量genymdhms(生成日期，年、月、日、时、分、秒)…
    可以替换为generationTimestamp
5. 使用可搜索的名称

单字母名称和数字常量有个问题，就是很难在一大篇文字中找出来。找MAX_CLASSES_PER_STUDENT很容易，想找7或者e就麻烦了。

单字母名称仅用于短方法中的本地变量

名称长短应与其作用域大小相对应

6. 避免使用编码
  1. 匈牙利语标记法
  2. 成员前缀
  3. 接口和实现
7. 避免思维映射

不应当让读者在脑中把你的名称翻译为他们熟知的名称

8. 类名

类名和对象名应该是名词或名词短语，Customer、WikiPage
避免使用Manager、Processor、Data或Info这样的类名
类名不应当是动词

9. 方法名
  1. 方法名应当是动词或动词短语，eg：postPayment、deletePage
  2. 属性访问器、修改器和断言应该根据其值命名，并根据javabean标准加上set、get和is前缀
10. 每个概念对应一个词

给每个抽象概念选一个词，并且一以贯之。

11. 别用双关语

eg：add有时可以用insert、append之类词替代更清楚。

12. 使用解决方案领域名称

给这些事取个技术性的名称

13. 使用源自所涉问题领域的名称
14. 添加有意义的语境
  1. 可以添加有意义的前缀，eg：addrFirstName、addrLastName、addrState
  2. 创建名为Address的类
15. 不要添加没用的语境

对Address类的实体来说，accountAddress和customerAddress都是不错的名称，不过用在类名上就不太好了。Address是个好类名。如果需要与MAC地址、端口地址和Web地址相区别，我会考虑使用PostalAddress、MAC和URL。
这样的名称更为精确，精确正是命名的要点。

二、函数

1. 短小

函数的第一规则是要短小

作者说，一个函数每行不应该超过150个字符，20行封顶最佳。

2. 只做一件事

函数应该做一件事。做好这件事。只做这一件事。
用TO起头段落来描述这个函数

3. 每个函数一个抽象层级

自顶向下读代码：向下规则
逻辑树分析，一层一层向下写

4. 使用描述性的名称
5. 函数参数
没有参数最好，一个参数还行，两个参数勉强，三个参数避免
